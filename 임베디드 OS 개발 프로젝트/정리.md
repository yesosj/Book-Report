1.6 하드웨어 정보 읽기
- 펌웨어 개발은 결국 하드웨어의 정보를 읽고 쓰는데 이때 레지스터 이용. 이는 hw, sw 상호작용하는 인터페이스임.
So , 펌웨어 개발 시 그 hw의 레지스터 사용법을 알아함. Hw의 레지스터 사용법은 데이터 시트에 나옴.
So, 펌웨어 개발자는 자신이 개발하는 대상 hw의 데이터시트를 항상 보며 만들어야함.
- Entry.s를 수정하여 LDR R0, =0x10000000 은 R0에 0x10000000 숫자 넣으라는 것이고,LDR R1, [R0]는 R0에 저장된 메모리 주소(0x100000000)에서 값을 읽어 R1에 저장하라는 것 . 대체 여기 주소에는 뭐가 있나? 그걸 알려면 realviewPB의 데이터시트를 봐야함 . 이는 arm에서 파는 물건인 만큼 arm사 에서 운영하는 웹사이트인 인포센터에서 데이터 시트 구할 수 있음. 찾아보니 ID register라고 나옴 이런 레지스터에서 고윳값을 읽고 하드웨어를 구분하는 것임. 여기에도 어떤 고윳값이 들어가야함. 이는 sys_id라고 불리고 fpga, arch, build, hbi, rev로 나누어짐. 각각 8,4 4 12 4 비트씩 할당됨. Rev는 보드 버전, hbi는 보드 번호, build는 보드 빌드 변형, arch는 버스 아키텍처 의미. 어렵기 때문에 예제를 따로 제공하는 애플리케이션 노트라는 것을 제조사가 제공하기도 함.

- make
- make debug
- make gdb
(gdb)
- target remote:1234 // qemu와 gdb 연결
- file build/navilos.axf // axf파일 로드
- list // 현대 디버깅 중인 프로그램의 소스코드
- info register// qemu에서 실행 중인 가상 cpu의 레지스터 값
- s // step으로 한줄 실행 후 정지, 함수내부로 진입. 즉 list에 있는 코드에서 vector start 다음줄을 첫번째 줄로 인식하여 실행.
- n // 한 줄 실행 후 정지, 내부진입안함

- s 를 암튼 실행하면 r0에 입력되고 다시 s 하면 r1에 값이 들어감. 이때 들어간 이값이 해당 realviewpb의 데이터시트의 레지스터주소 0x10000000에 해당 들어가있는 값임. 그값이 결국 0x1780500이고 이를 이진수로 바꾸면 결국 앞서 말한 Sys id 레지스터 값이 나옴.

=> 위의 방법이 결국 계속 반복됨. 즉 hw에서 제대로 동작하는 지 확인해보려고 hw의 레지스터에 접근해 봤으며 결과를 확인하려고 gdb도 써봄. 이때 r0, r1 은 cpu의 레지스터 이고 0x 어쩌구는 전부 하드웨어의 레지스터 정보임.
Qemu는 가상화된 하드웨어 환경을 제공하는 sw 이고 realviewpb는 실제 하드웨어임.

3. UART
- QEMU에서 사용가능한 하드웨어 중 첫번째는 UART
- Realviewpb에서 uart하드웨어의 명칭은 pl011이고 가장 먼저 할 일은 uart의 레지스터를 코드로 만드는 일.
- 꽤 많은 레지스터중 일부만 코드화시키는 데 방법은 2가지. 첫번짼 C언어 매크로 이용 ,구번짼 구조체 이용. 취향차이임.
- hal 폴더 안에 rvpb안에 uart.h파일에 작성하고 rasppi나 다른 하드웨어를 추가하는 것도 가능ㅎ
- 즉 다양한 종류의 uart 하드웨어가 존재할 수 있고 이는 각자의 방식으로 동작하더라도 이들을 사용하는 코드은 공용 인터페이스를 통해서 같은 방식으로 사용할 수 있어야함. 즉 일종의 디바이스 드라이버가 필요함-> so, uart api 가 관리함. 이런 공용 인터페이스 혹은 api 설계를 HAL(hardware abstraction layer) 라고 함. 이때 이 파일명은 HalUart.h 이고 hal 바로 밑에 위치함. 이유는 같은 위치에 rvpb, rasppi 등의 각 펌웨어 코드가 존재하므로 uart를 공통으로 접근하려면 동급으로 위치해야함.
- 해당 공용 인터페이스(디바이스 드라이버) 내용은 uart 를 초기화하고 출력 테스트하는 함수만 정의.
그리고 각 하드웨어 hal 구현체에서 이 haluart.h파일을 #include로 추가하여 각 하드웨어에 맞춰서 구현함.

4. 인터럽트
- 컴퓨팅 시스템의 꽃으로, 인터럽트를 처리하려면 우선 인터럽트 컨트롤러를 어떻게 사용해야 하는 지 알아야함. 그 다음 인터럽트 컨트롤러를 초기화하고 사용하는 코드를 작성해야함. 인터럽트 컨트롤러 관련 코드를 다 작성했으면 실제 인터럽트를 발생시키는 하드웨어와 인터럽트 컨트롤러를 연결해야함. 현재 uart하드웨어는 인터럽트를 발생시키므로 uart 하드웨어와 인터럽트 컨트롤러를 연결함. 그러면 uart하드웨어가 인터럽트 컨트롤러로 인터럽트 신호보냄. 인터럽트 컨트롤러는 arm코어로 인터럽트 보냄. 펌웨어에서 cpsr의 irq, fiq 마스크를 끄면 irq, fiq가 발생했을 때 코어가 자동으로 익셉션 핸들러를 호출해야함. 이때는 익섹션 핸들러는 펌웨어임. 그러므로 이것도 작성해야함.
- 인터럽트 작동 여부를 확인하기 위해 main함수에 무한 루프를 만들고 키보드를 입력했을 때 이에 반응하도록 코드를 작성해보겠습니다.
- 인터럽트.c 코드를 작성하는데 인터럽트를 켜고 끄는 기능, 인터럽트 핸들러에 대한 처리 방법
- cspr 을 제어하려면 Entry.S 처럼 어셈블리어 소스 파일을 만들거나 c언어에 인라인 어셈블리어를 하는 방법존재.

4.1 인터럽트 컨트롤러
- GIC(범용 인터럽트 컨트롤러) ARM용.
- GIC작성 후 이걸로는 실제 인터럽트를 활용못함. 이유는 인터럽트를 발생시키는 하드웨어와 연결되어있지 않음. 즉 인터럽트를 받고 처리할 준비가 됐으므로 하드웨어를 설정해서 인터럽트 컨트롤러로 인터럽트를 보내야 활용가능. 역시 가장먼저 작업할 하드웨어는 UART임.가장 많이 쓰는 입력 인터럽트.

4.2 UART와 인터럽트 연결

- 인터럽트 컨트롤러를 초기화하는 함수를 uart 초기화 함수보다 먼저 호출해야함.

4.3 irq 익셉션 벡터 연결
- 인터럽트가 벌생하면 인터럽트 컨트롤러는 이 인터럽트를 접수해서 arm코어로 바로 전달함. 앞서 말했듯 arm에 fiq, irq라는 두 종류의 인터럽트가 있음. 여기선 irq만 사용. So, arm코어는 인터럽트를 받으면 irq익셉션를 발생시킴. 그리고 동작모드를 irq모드로 바꾸면서 동시에 익셉션 벡터 테이블의 irq 익셉션 벡터로 바로 점프함. So, 하드웨어동작을 빼고 소프트웨어 동작만 보면 인터럽트 종류가 무엇이든 일단 익셉션 벡터 테이블의 irq 익셉션 핸들러 무조건 실행. 남은 작업은 익셉션 벡터 테이블의 irq 익셉션벡터와 인터럽트 컨트롤러의 인터럽트 핸들러를 연결함.
- 그 전에 익셉션 핸들러 작성. About irq, fiq

9. 이벤트
- 버튼을 누르면 프로그램 내부에서는 버튼을 눌렀다는 이벤트가 발생하고 이벤트 핸들러로 전달됨. 어떤 임베디드 시스템에 있는 버튼을 사용자가 누르면 시스템 내부적으로는 버튼에 연결된 스위치에서 전기가 연결됨. 컨트롤러는 해당 전기 신호를 인식함. 그러면 물리적전기신호를 소프트웨어적인 인터럽트로 처리함. Arm이라면 irq, fiq가 발생. 앞장에서 설명한 것과 겉이 irq, fiq의 핸들러에서 인터럽트 컨트롤러의 레지스터를 읽어서 어떤 인터럽트인지 확인함.

11. 동기화
- 나빌로스 프로젝트만 보면 비선점형 스케줄링인데다가 싱글 코어 환경이므로 동기화는 필요없다.
But 운체에선 매우 중요한 부분

- 멀티코어 롼경에서 테스크간 동기화
- 동기화란 운체에서 어떤 작업을 아토믹 오퍼레이션으로 만들어준다는 의미. 아토믹 이란 해당 작업이 끝날 때까지 컨텍스트 스위칭 발생이 없다는 의미. 이때 해당 작업을 크리티컬 섹션이라고 부름.
- 즉 동기화란 어떤 작업이 크리티컬 섹션이라고 판단될 경우, 해당 크리티컬 섹션을 아토믹 오퍼레이션으로 만들어 주는것 == 상호배제(mutual exclusion)
- 구현 방법은 1. 세마포어 2. 뮤택스 3. 스핀락

11.1 세마포어
- 가장 유명하고 좋은 동기화방법
- 이는 가장 간단한 함수 두개로 구현되므 test와 release. Test는 크리티컬 섹션에 진입 가능한지 확인하는 함수(lock) , release는 크리티컬 섹션을 나갈때 호출해서 세마포어를 놓아주는 혹은 세마포어의 잠금을 푸는역할(unlock)
- 종류 : 이진 세마포어, 카운팅 세마포어, 뮤택스

11.2 뮤택스
- 이는 바이너리 세마포어의 일종
- 세마포어는 잠금에 대한 소유 개념이 없으므로 누가 잠근 세마포어이든 간에 누구나 잠금 풀 수 있음. 그러나 뮤택스는 소유개념이 존재하여 뮤택스를 잠근 테스크만이 잠금을 풀 수 있음.
- 즉 뮤택스는 바이너리 세마포어에 소유의 개념을 더한 동기화 알고리즘임.
- 세마포어는 잠금장치가 없는 화장실, 뮤택스는 있는 화장실.

11.3 스핀락
- 바쁜 대기 개념의 크리티컬 섹션 보호 기능. 바쁜 대기란 스케줄링을 하지 않고 cpu를 점유한 상태, 즉 cpu가 여전히 바쁜 상태에서 락이 풀리는 것을 대기함. 스케줄리을 하지 않고 짧은 시간 동안 cpu를 점유하면서 잠금이 풀리는 것을 기다린다는 아이디어이므로 멀티코어 환경에서 유용하게 쓰임.

13.1 익센션 벡터 테이블
- 전원이 켜지면 arm은 익센션벡터테이블을 읽음. 이 위치를 조정가능하지만 대부분은 안바꾸고 arm이 정한 기본 위치인 0x0000000메모리 주소에 배치함. 여긴 시작위치. 그래서 전원이 켜지면 여기부터 시작.
- 익셉션은 총 7개 존재. 중간 0x14오프셋은 사용 안함. Arm은 익셉션 벡터테이블에 정의된 상황이 발생하면 프로그램 카운터를 익셉션 벡터 테이블에 정의된 오프셋 주소로 강제 변환. 그리고 테이블에 있는 명령 바로 실행.익셉션은 각 각 4바이트 할당. 32비트 머신이니깐 한 익셉션에 명령어 한 개만 실행가능. 상식적으로 꽤 많이 명령어가 필요할텐데 왜 딱 한개만 넣을 수 있는 공간만 배정했을까? 그것은 익셉션 처리에 필요한 코드가 개발자마다, 시스템마다, 펌웨어마다 다 다르기 때문. 그래서 특정한 크기를 예약해 놓으면 어떤 사람은 그 크기가 부족하고 어떤 사람은 그 크기가 남아서 소중한 메모리자원낭비하는 문제 발생. So, 명령어를 쓸수 있는 최소 크기(4바이트) 를 배정하고 거기에 브랜치 명령어를 써놔서 익셉션을 처리하는 코드로 점프. 이를 익셉션 핸들러라고 함.
- 정리하면 익셉션이 발생하면 arm코어는 pc 를 익셉션 벡터 테이블의 익셉션 벡터 오프셋으로 변경. 익셉션 벡터 오프셋의 크기는 명령어 한개만 쓸수 있는 크기. 거기에 브랜치 명령을 쓰고 브랜치 명령에 따라 다시 점프하면 익셉션 핸들러로 진입. 이 익셉션 핸들러에서 익셉션을 처리.
- 익셉션 : 주변장치(peripheral) 등에서 발생한 신호를 처리하기 위해 순차적으로 진행하던 프로그램의 수행을 끊는 것. 프로그램 수행을 끊는 것은 단순히 pc의 값을 바꾸는 것과 조금 다른 의미임.

- 프로그램 상태 레지스터 : cpsr, spsr
