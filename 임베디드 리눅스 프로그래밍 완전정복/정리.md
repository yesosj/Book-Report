7. 문자 장치 (Character Device)

문자 장치는 사용자 공간에서 장치 노드라는 특수 파일로 식별 가능

해당 파일명은 **주번호(Major)와 부번호(Minor)**를 이용해 장치 드라이버에 매핑됨.

주번호 → 장치 노드를 특정 장치 드라이버에 매핑

부번호 → 드라이버에 어떤 인터페이스가 액세스되는지 알려줌

예시

첫번째 시리얼 포트

/dev/ttyAMA0
- 주번호 : 204
- 부번호 : 64

두번째 시리얼 포트

- 동일한 장치 드라이버에 의해 처리됨
- 동일한 주번호
- 부번호 : 65

확인 명령어


ls -l /dev/ttyAMA*


→ 모든 주번호와 부번호 확인 가능

또한 표준 주번호와 부번호 목록은 아래 위치에 정의됨


Documentation/devices.txt
drivers/tty/serial/amba-p1011.c

에 정의됨.

8. 장치 노드 생성 방법

장치 노드는 여러 가지 방법으로 생성 가능

## 1. devtmpfs

드라이버에서 제공하는 기본 이름과 인스턴스 번호를 사용해  
장치 드라이버가 새로운 장치 인터페이스를 등록할 때 생성되는 코드

## 2. udev / mdev

기본적으로 devtmpfs와 동일하지만

사용자 공간 데몬 프로그램이  
**sysfs에서 장치명을 추출해 노드를 생성해야 한다는 점이 다르다**

## 3. mknod

정적 장치 노드는 **이 명령어로 수동 생성**

# 13. 네트워크 장치 확인

네트워크 장치는 **장치 노드를 갖고 있지 않기 때문에 /proc/devices 목록에 나오지 않는다.**

대신 아래 명령어 사용

ifconfig, ip link show

lspci
→ USB, PCI 버스에 연결된 장치 확인 가능


Sysfs

사실 정말 재미있는 녀석이다.
이는 커널 객체, 속성 관계를 대표하는 것이라 말할 수 있다. 이때 커널 객체는 디렉터리고 속성은 파일이며 관계는 한 객체를 다른 객체로 연결하는 심볼릭 링크다.
ls /sys
1. /sys/devices
- 이는 부팅 이후 발견한 장치들에 대한 커널의 관점과 이들이 서로 어떻게 연결돼 있는지를 보여준다.
-공통적으로 갖고 있는 디렉터리는 다음 세 개이다
- system/ : 이는 cpu와 클럭을 비롯한 시스템의 핵심 장치들에 대한 정보를 포함
- virtual/ : 이는 메모리 기반의 장치에 대한 정보를 포함하고 virtual/mem 에서 /dev/null, random, zero로 타나타는 메모리 장치들에 대한 정보를 찾을 수 있고 루프백 장치에 대한 정보는 virtual/net에서 찾을 수 있다.
- platform/ : 이는 기존의 하드웨어 버스에 연결되지 않은 모든 장치에 대한 정보를 보여줌. 임베디드 장치 대부분이 해당됨.
이외의 장치들은 각각의 실제 시스템버스에 해당하는 디렉터리 정보를 찾을 수 있지만 전부 탐색하는 일은 어렵다. 경로명이 길고 암기해야돼서. 따라서 /sys/class, block 이 다른 관점을 제시함.

2. /sys/class
- 이는 장치 드라이버의 형태에 따라 정리함. 즉 소프트웨어적인 관점보단 하드웨어적 관점에서의 목록이고 각 서브디렉터리는 각각의 드라이버의 클래스를 나타내며 이는 드라이버 프레임워크 컴포넌트가 구현한 것임. 예로 UART는 TTY계층에서 관리하고 /sys/class/tty 에서 확인 가능. 이런 방식으로 네트워크 장치는 /sys/class/net, 입력 장치는 /sys/class/input 으로 찾을 수 있음.
- 이와 같은 서브디렉터리내에는 /sys/deviec에 있는 해당 유형의 장치의 인스턴스들로 연결하는 심볼릭 링크가 존재함.
- ls /sys/class/tty/ttyAMA0 를 보면 여러 속성값을 볼 수 있고 dev를 보면 주와 부번호를 알 수 있다

3. /sys/block
- 여기에는 각 블록 장치의 서브디렉터리가 존재함.
만약 비글본 블랙이라면 이 보드의 eMMc칩인 mmvblk1 을 살펴보면 해당 인터페이스의 속성과 내부 파티션을 확인할 수 있다.
- ls /sys/block/mmcblk1
결론적으로 시스템에 존재하는 장치(하드웨어)와 드라이버(소프트웨어)에 대해선 sysfs을 읽어보면 많은 것을 배울 수 있다.

적합한 장치 드라이버 찾기
- 전현적인 임베디드 보드는 특정 애플리케이션에 적합하도록 변경한 제조사의 래퍼런스 설계를 기반으로 하는 것이 대부분. 래퍼런스 보드와 함께 제공되는 bsp는 해당 보드의 모든 주변기기를 지원해야하지만, I2C를 이용해 붙인 온도 센서나 gpio핀에 연결된 전구와 버튼, mipi인터페이스와 연결한 디스플레이 패널 등의 다양한 외장 장치를 추가해 직접 디자인 가능.
- 이런 주변 기기릉 모두 지원하는 장치 드라이버를 어디서부터 찾아봐야 할까.가장 먼저 제조사 웹 사이트의 드라이버 지원 페이지를 찾아보거나 혹은 제조사이 직접 문의해볼 수 있다. 하지만 이런 경우 하드웨어 제조사가 리눅스에대해 잘 몰라 잘못된 정보를 주기도함. 우리가 필요로 하는 커널과 다른 버전일 수도 있음.
그러므로 꼭 이 방법을 시도하자.
작업중인 건에 대해서는 항상 오픈소스 드라이버를 찾아보자. Make menuconfig 에서 일반적인 제품명으로 검색해보기.
다음으론 drivers 디렉터리의 코드를 검색해보기
(Grep으로가능)
만약 못찾으면 직접 작성하는 데 정말 작성할 필요가 있는 지 또 생각하라.

커널 코드 작성없이 사용자 공간에서 직접 하드웨어에 접근할 수 있는 방법도 있음.
사용자 공간의 장치 드라이버는 크게 두가지이다
1. Gpio와 led처럼 sysfs의 파일을 통해 제어하는 방식
- ls /sys/class/gpio/gpiochip96
- echo 53 > /sys/class/gpio/export
: 비글본 블랙의 사용자 led0에 연결된 gpio 53에서 이를 진행한 것임.

Gpio가 인터럽트를 생성 가능한 상태면 edge라는 이름의 파일이 존재하고 이는 rising, falling, both, none으로 나뉨.
Led는 보콩 gpio핀을 통해 제어하는 경우가 대부분.
더 자세한 정보는 Documentaion/leds에서 확인가능하며 해당 드라이버는 drivers/leds에서 찾을 수 있음.
ls /sys/class/leds

2. I2C처럼 장치 노드를 통해 공통 인터페이스를 노출시키는 시리얼 버스를 통해 제어하는 방식.

- I2C는 임베디드 보드에 흔히 탑재돼 있는 단순한 형태의 저속 2선식 버스로, 보통 디스플레이 컨트롤러나 카메라 센서, GpIo확장 모듈같이 SoC보드에 설치돼 이씨 장ㄴㅎ은 주변기기에 접근할 때 사용한다. 이와 관련된 표준으론 PC에서 온도와 전압 센서에 접근할 때 사용하는 SMBus가 있다. 이는ㅑ
마스터 슬레이브 형식임.

또다른 인기있는 통신 프로토콜은 4선 버스인 spi이다. 이는 i2c와 비슷하지만 훨씬 빠른 속도를 보인다. 이러한 장치 드라이버는 모두 일반 드라이버(gpio, led, i2c, spi)이므로 사용자 공간에서 엑세스하는 것은 간단했다. 하지만 어느 시점에선 호환되는 커널 장치 드라이버가 없는 하드웨어를 보게될것이다. (Lidar, sdr) soc와 이 하드웨어 사이에 fpga가 있을 수도 있다. 이러한 상황에서는 자신의 커널 모듈을 직접 작성하는 것 이외에 다른 방법은 없다.

Read(2), write(2) 외에 장치 드라이버와 의사소통할 수 있는 방법이 감사하게도 존재함.

Read , write 외에 장치 드라이버와 소통하는 방법
1. ioct1 : 이 함수는 드라이버에 2개의 인자를 넘길 수 있으며 첫번째는 드라이버 여러 함수중 하나를 선택하는 명령어, 두번째는 입력과 출력 매개 변수를 위한 컨테이너 역할을 하는 구조체의 포인터를 사용하는 경우가 많다. 하지만 이는 커널 코드와 애플리케이션 코드를 너무 의존적으로 묶어놔서 개발자가 선호하지 않음.
2. Sysfs : 최근에 선호하는 방식. 스크립트로 활용 가능, 하지만 한가지 값만가져야하는 단점. Icot1은 하나의 함수호출에도 모든 인자 전송가능.
3. Mmap: 이는 커널을 사용하지 않고도 커널 메모리를 사용자 공간으로 매핑해 커널 버퍼와 하드웨어 레지스터로 직접 접근할 수 있다. 인터럽트와 dma를 제어하려면 여전히 커널 코드를 작성해야할 필요가 있지만 .
4. Sigio : 커널의 함수인 kill fasync 를 이용하면 입력 장치가 준비되고나 인터럽트를 받는 등의 이벤트 때 드라이버에서 앱으로 신호를 보내 이를 알 릴 수 있다. 다만 사용자 공강에서 안정적인 신호 처리기를 만들기 어렵다는 문제점 때문에 사용하는 사람은 적음.
5. Debugfs : 이는 proc 및 sysfs 와 유사한 방식으로 커널 데이터를 파일과 디렉터리로 표현하는 의사 파일시스템이다. 하지만 둘과 다르게 어직 디버깅과 추적 정보를 위한 것이므로 시스템의 일반 작업에 필요한 정보는 포함하지 말아여한다
6. Proc : ps와 연관되지 않는 한 새 코드레서는 사용되지 않는다. Sysfs, debugfs 와 다르게 이는 non-gpl모듈에서도 사용가능
7. Net link: 소켓 프로토콜 일종인데 거의 사용되지 않음.

드라이버 코드에 얼마든지 재미있는 인터페이스를 설계할 수 있다. 다만 놀람 최소화 원칙을 지켜야함

장치드라이버의 구조

- /dev/dummy0 부터 /dev/dummy3까지 4개의 장치를 생성하는 dummy라는 이름을 가진 장치 드라이버 코드의 시작부분이다. 다음으론 캐릭터 장치 인터페이스에 대항 dummy_open, dummy_release, dummy_read, dummy_write 함수를 정의함
그후, file operations 구조를 초기화하고 드라이버가 로드 언로드될 때 호출되는 dummy init 과 dummy exit 함수를 정의함.
코드 끝에 module init과 module exit 라는 매크로는 모듈이 로드, 언로드될 때 호출될 함수를 지정함. 마지막에 있는 MODULE_*라는 이름의 새 매크로는 모듈에 대흔 몇가지 기본 정보를 추가한 것.
MODULE_LICENSE("GPL")
MODULE_AUTHOR("Chris Simmonds")
MODULE_DESCRIPTION("A dummy driver")

커널 모듈 컴파일하기
Makefile 을 수정하여 obj-m : = dummy.o 이는 dummy.c 파일을 가져온 후 dummy.ko라는 커널모듈을 생성함. 드라이버가 기본 문자 장치라면 dummy.c 파일을 drivers/char에 넣으면됨.
무조건 내장 요소를 빌드하고 싶다면
Obj-y += dummy.o 추가

만약 드라이버를 선택할 수 있도록 만들고 싶다면 Kconfig 파일에 메뉴 옵션을 추가하고 그성 옵션에 컴파일 조건을 만들면 됨.

커널 모듈 로딩
insmod, lsmod, rmmod
depmod -a 로 모듈의 의존성 데이터베이스를 생성하자.
module.* 파일 내의 정보는 modprobe로 모듈을 경로명이 아닌 이름으로 찾을 때 사용 가능.

표준이 아닌 장비 사용 시 직접 드라이버를 만들어야함. Gpio, i2c 등을 위한 기본 드라이버를 사용하고 사용자 공간 코드를 작설해 이러한 역할을 하게 만들어서 이러한 문제를 회피하는 방법도 있음. 커널코드를 작성하지 않고도 하드웨어에 익숙해질 필요있음. 드라이버 코드를 작성하다 디버깅하고 싶다면 gdb로 하기

/proc/devices : 현재 로드돼 동작 중인 문자와 블록 장치 드라이버 목록을 볼 수 있다.

장치 드라이버를 직접 작성하기 전에 정말로 작성할 필요가 있는지를 생각해볼 시간이 필요함. 커널 코드를 직접 작성할 필요없이 사용자 공간에서 직접 하드웨어에 접근할 수 있는 방법도 있음.

사용자 공간의 장치 드라이버는 크게 두가지임.

첫번째 GPIO와 LED처럼 sysfs의 파일을 통해 제어하는 방식.
두번째, I2C처럼 장치 노드를 통해 공통 인터페이스를 노출시키는 시리얼 버스를 통해 제어하는 방식.

GPIO : 2개의 상태값 HIGH, LOW 중 하나를 갖는 하드웨어 핀에 직접 접근할 수 있는 가장 단순한 형태의 디지털 인터페이스. 보통 푸시 버튼과 led, 모터 등 가장 많이 쓰임

I2C : 임베디드 보드에 흔히 탑재돼 있는 단순한 형태의 저속 2선식 버스로, 일반적으로 디스플레이 컨트롤러나 카메라 센서, GPIO확장 모듈과 같이 SoC보드에 설치돼 있지 않은 주변기기에 접근할 때 사용

SPI : I2C와 비슷하지만 최대 수십 MHz 정도의 빠른 속도를 보임

사용자 공간 옵션들에 지칠 때쯤 결국은 장치에 연결된 하드웨어에 접근하게 위한 장치 드라이버를 작성하는 지경에 이를 것이다. 문자 장치은 가장 융통성이 있으므로 90%정도는 이를 사용하면 될 것이고 네트워크 인터페이스 작업이 필요하면 네트워크 장치를 사용하고 대용량 저장 장치의 경우 블록 장치를 사용하면 된다. 커널 드라이버를 작성하는 작업은 복잡하고 이 책의 범위를 넘는 일이다



# Sysfs

사실 정말 재미있는 녀석이다.

이는 **커널 객체, 속성 관계를 대표하는 것**이라 말할 수 있다.  
이때

- 커널 객체 → 디렉터리  
- 속성 → 파일  
- 관계 → 한 객체를 다른 객체로 연결하는 심볼릭 링크  

명령어
ls /sys


# 1. /sys/devices

이는 **부팅 이후 발견한 장치들에 대한 커널의 관점과 이들이 서로 어떻게 연결돼 있는지**를 보여준다.

공통적으로 갖고 있는 디렉터리는 다음 세 개이다.

### system/

- cpu와 클럭을 비롯한 **시스템의 핵심 장치들에 대한 정보 포함**

### virtual/

- 메모리 기반 장치 정보 포함
- `virtual/mem` 에서 `/dev/null`, `random`, `zero` 같은 메모리 장치 정보 확인 가능
- 루프백 장치 정보는 `virtual/net` 에서 확인 가능

### platform/

- 기존 하드웨어 버스에 연결되지 않은 장치 정보
- **임베디드 장치 대부분이 해당**

이외의 장치들은 각각 실제 시스템 버스에 해당하는 디렉터리에서 찾을 수 있지만  
경로명이 길고 암기하기 어렵다.

따라서
/sys/class
/sys/block


이 다른 관점을 제시한다.

---

# 2. /sys/class

이는 **장치 드라이버의 형태에 따라 정리**한다.

즉 **소프트웨어적인 관점보다는 하드웨어적 관점에서의 목록**이다.

각 서브디렉터리는 각각의 **드라이버 클래스**를 나타낸다.

예

- UART → TTY 계층에서 관리
- `/sys/class/tty` 에서 확인 가능

다른 예

/sys/class/net
/sys/class/input


- 네트워크 장치
- 입력 장치

또한 이 서브디렉터리 안에는 /sys/devices 에 있는 장치 인스턴스로 연결되는 **심볼릭 링크**가 존재한다.

예

ls /sys/class/tty/ttyAMA0 여기서 여러 속성 값을 볼 수 있으며 
dev 파일을 보면 **주번호와 부번호**를 확인할 수 있다.

---

# 3. /sys/block

여기에는 **각 블록 장치의 서브디렉터리**가 존재한다.

예

비글본 블랙의 eMMC 칩 mmcblk1   
명령어
 ls /sys/block/mmcblk1


이를 통해

- 인터페이스 속성
- 내부 파티션

을 확인할 수 있다.

결론적으로

**시스템에 존재하는 장치(하드웨어)와 드라이버(소프트웨어)에 대해 sysfs를 읽어보면 많은 것을 배울 수 있다.**

---

# 적합한 장치 드라이버 찾기

전형적인 임베디드 보드는 특정 애플리케이션에 적합하도록 변경한 **제조사의 레퍼런스 설계**를 기반으로 한다.

레퍼런스 보드와 함께 제공되는 **BSP**는 해당 보드의 모든 주변기기를 지원해야 한다.

하지만 다음과 같은 **외장 장치**를 직접 추가할 수도 있다.

- I2C 온도 센서
- GPIO 전구
- 버튼
- MIPI 디스플레이 패널

이런 장치들을 지원하는 드라이버는 어디서 찾을까?

### 방법

1. 제조사 웹사이트 드라이버 페이지 확인
2. 제조사 문의

- 커널 버전이 다를 수도 있다

따라서 다음 방법을 시도하자.

### 오픈소스 드라이버 찾기

- `make menuconfig` 에서 제품명 검색
- `drivers` 디렉터리 검색
- `grep` 사용

그래도 없다면 **직접 작성**

**정말 작성할 필요가 있는지 먼저 생각해야 한다.**
