# 2. 프로세스

프로세스는 유저레벨과 커널레벨에서 존재
유저에선 glib 도움으로 커널에게 프로세스 생성 요청.--> init 프로세스가 생성
커널에선 kthread_create 함수를 호출해서 커널 프로세스 생성--> kthreadd 프로세스가 생성
둘의 흐름은 다르나 공통점은 프로세스를 생성할 때 do fork 함수를 호출한다는 점.

사실 프로세스는 생성이 아니라 복제됨. 생성은 시간이 오래걸림.

유저모드에선 sys_clone시스템 콜 핸들러 함수가,
커널모드에선 kernel_thread함수가 do fork 를 호출함.

* ps -ejH : 부모와 자식 관계로 유저와 커널 스레드 종류 목록 확인 가능
커널스레드는 데몬과 비슷하지만 유저영역과 싯.템 콜을 받지 않고 동작하는 것에서 다름.

* 데몬 : 백그라운드오 os에소 실행되는 프로세스 보통 시스템의 지속적인 작업에 사용
* idle : 각 cpu마다 무조건 1개 존재하는 커널 스레드 pid=0임. Cpu가 할 일이 전혀 없을 때 실행되는 더미 스레드
* kworker: workqueue를 처리하는 백그라운드 작업 전용 커널 스레드. 커널이 미뤄던 작업을 대신 비동기로 처리.

-->실행 가능한 task가 생기면 idle은 즉시 중단되고 새 테스트(kworker)가 실행됨.

=> 결론 : task_struct(테스크 디스크립터)에는 식별(comm, pid), 상태(state, flags), ps간 관계(parent, sibling, children), 실행시각(utime stime) 정보인 프로세스의 속성 정보를 가진다.
반면, thread_info 에는 핵심정보인 선점 스케줄링 실행 여부, 시그널 전달 여부, 인터럽트 컨텍스트와 soft irq컨텍스트 상태, 휴면 상태로 진입하기 직전 레지스터 세트를 로딩 및 백업과 세부 실행정보인 컨텍스트 정보, 스케줄링 직전 실행했던 레지스터 세트, 프로세스 세부 실행 정보와 같은 프로세스의 실행동작을 관리한다. 프로세스마다 1개의 thread_info 존재. BUT CPU마다 thread info 구조체의 필드가 다르고 구현방식이 다르다.

=> task_struct 과 thread info 차이점은 전자는 cpu에 독립적으로 어떤 cpu에서나 기본 필드가 같지만, 후자는 cpu아키텍처에 종속적인 프로세스의 세부 속성을 저장하므로 서로 다르다.

프로세스의 속성 정보를 표현하는 가장 중요한 자료구조는 테스크 디스크립터를 나타내는 task struct 구조체.
해당 pid들의 대한 task struct은 /proc 에 있고 status보면 기본 정보 알수있음

Thread info는 프로세스의 세부 실행 정보 저장 및 로딩하는 자료구조 . 이는 프로세스 스택의 최상단 주소에 위치. 프로세스마다 자신만릐 스택이 있으니 프로세스마다 1개의 thread info 구조체 가짐.

이때 중요한 task struct 과 thread info 차이는
Task struct은 커널 버전이 같으면 x86이나 arm 에서 task struct 구조체의 기본 필드는 같음. 그러나 thread info 구조체는 cpu 아키텍처에 종속적린 세부속성을 가지므로 서로 다름.

따라서 task_struct 안에 **thread_info**가 존재한다는 것은 아니고, 각각 다른 메모리 위치에 존재하지만 서로 밀접하게 연결되어 있다는 것입니다.

정리

Ps 생성 시 커널로부터 스택 공간을 할당 받는 데 32비투 에선 0x2000, 64비트에성 0x4000 스택을 받음ㅁ.

# 3. 인터럽트

=> 결론 : 인터럽트 발생(익셉션발생)->익셉션 벡터-->인터럽트 벡터(vector_irq)실행->(irq_svc: 커널모드 or irq_usr : 유저모드)-> (실행중인 프로세스의 레지스터 세트를 프로세스 스택에 저장) -> 인터럽트에 대응하는 인터럽트 디스크립터(irq_des) 읽기 -> 인터럽트 종류별로 인터럽트 핸들러(handle_irq_event_percpu) 호출 -> (인터럽트 컨텍스트 시작) -> 인터럽트 핸들러의 서브루틴 실행 시작 -> 인터럽트 핸들러의 서브루틴 실행 마무리 -> (인터럽트 컨텍스트 마무리)

: 실행 중인 코드가 인터럽트 컨텍스트인지 알려면 in_interrupt()함수를 호출해서 true면 인거임.

* 인터럽트 디스크립터
: 인터럽트의 세부 속성 관리 자료구조. 인터럽트 핸들러의 주소 정보 가짐.

* 인터럽트 컨텍스트
: 현재 실행중인 프로세스가 현재 인터럽트를 처리중이라는 의미. 컨텍스트란 프로세스 실행 구 자체를 의미.

* 프로세스 실행 정보는 레지스터 세트인cpu_context_save 구조체에 저장되어있고 이는 thread info 의 cpu context 필드에 저장됨

[하드웨어 관점에서]
인터럽트란 하드웨어의 변화를 감지해서 외부 입력으로 전달되는 전기 신호. 하드웨어 개발자들은 오실로스코프장비로 인터럽트 싱호가 제대로 올라오는 지 측정함.
[스프트웨어 관점에서]
인터럽트는 하던 일을 멈추고 이미 정해진 코드를 실행해서 하드웨어의 변화를 처리함. 이때 정해진 코드란 인터럽트 벡터와 인터럽트 핸들러 의미. 이때 인터럽트가 발생하면 소프트웨어적으로 처리하는 과정을 인터럽트 서비스 루틴이라고 함.
[Cpu관점에서]
인터럽트는 cpu 아키텍처별로 다르게 처리함. Arm7 아키텍처에서는 외부 하드웨어 입력이나 오류 이벤트가 발생하면 익셉션 모드로 진입함. 인터럽트나 소프트웨어적으로 심각한 오류가 발생하면 arm7 프로세서는 이미 정해진 주소에 있는 코드인 익셉션 벡터를 실행함. 인터럽트 핸들러는 빨리 실행해야함. 이유는 실행중인 코드가 멈추기 때문.

주요 개념
1. 인터럽트 핸들러
- 인터럽트가 발생하면 이를 핸들링하기 위한 함수가 호출되는데 이를 인터럽트 핸들러라고 함. 예로 키보드 타이핑해서 인터럽트가 발생하면 키보드 인터럽트를 처리하는 키보드 인터럽트 핸들러.
2. 인터럽트 컨텍스트
- 현재 실행 중인 코드가 인터럽트를 처리 중이라는 의미. 소프트웨어 관점에서 인터럽트 실행 흐름은 1. 프로세스 실행중 2. 인터럽트 벡터 실행 3 . 커널 인터럽트 내부 함수 호출 4. 인터럽트 종류별로 인터럽트 핸들러 호출 -> 인터럽트 컨텍스트 시작 5. 인터럽트 핸들러의 서브루틴 실행 시작 6. 인터럽트 핸들러의 서브루틴 실행 마무리 -> 인터럽트 컨텍스트 마무리. 현재 실행중인 코드가 인터럽트 컨텍스트인지 아는 방법은 in_interrupt() 함수를 호출하여 true면 실행중.
3. 인터럽트 디스크립터
- 인터럽트의 세부속성을 관리하는 자료구조, irq_desc 구조체임.

인터럽트가 발생하면 vector_irq 레이블이 가장 먼저 실행되고 그 레이블을 통해 __irq_svc(커널모드), 혹은 __irq_usr(유저모드) 레이블이 실행됨.

두 레이블이 가장 먼저하는 일은 실행중인 프로세스의 레지스터 세트를 스택 공간에 저장하는 것. 이유는 실행정보를 백업하기 위함.
Vector irq 레이블에거 irq svc 레이블을 브랜치하고 handle irq event percpu함수까지 실행함.
그 중간과장에서 generic handle irq함수가 호출한 irq to desc함수가 인터럽트 번호로 인터럽트 디스크립터의 주소를 반환함. 정리하면
1. Irq svc : 실행중인 레지스터세트 저장
2. 중간 과정 함수 : 인터럽트 예외 처리 수행
3. 그 중 irq to desc : 정수형 인터럽트 번호로 인터럽트 디스크립터를 읽어 인터럽트 핸들러 호출

인터럽트 등록?

=> 결론 : request_irq-> request_threaded_irq->

인터럽트 핸들러를 등록하는 코드는 실전 디바이스 드라이버 개발에서 만날 가능성이 높다. 호출하려면 우선 등록해야함. 등록은 인터럽트를 초기화하는 과정에 포함돼 있다.

* request irq 함수 사용법
: 초기화 시 호출&include linux interrupt.h 에 존재
- 역할 : 인터럽트를 제대로 등록하면 true, 오류가 발생하면 false 를 반환함. request threaded irq 함수를 호출하는 기능이 다임.

drivers/usb/host/dwc otg/dwc otg driver.c 를 보면 request irq 를 호출해 인터럽트를 초기화시킴.

* Request threaded irq : kernel/irq/manage.c 에 정의됨.
- 역할 : 인터럽트 번호로 인터럽트 디스크립터를 가져오기, irqaction구조체로 동적 메모리 할당하기, irqaction구조체 필드에 인터럽트 초기화 인자를 설정하기.

인터럽트 초기화 과정은 시스템이 부팅될 때 이뤄짐.

- 지금까지 request irq함수로 인터럽트 핸들러 등록하는 과정과 request threaded irq함수에서 인터럽트 디스크립터에 해당 인터럽트의 설정 정보를 저장하는코드에 대해 배움.

인터럽트 핸들러 등록 시 플래그 설정

- 보통 인터럽트 등록 후 호출하면 제대로 동작하지 않은 경우가 허다함. 따라서 오실로스코프로 파형을 측정해야하는데 이때 하드웨워 개발자와 분쟁이 생김.
1. 인터럽트를 발생시키는 데 하드웨어에 노이즈가 유입돼 인터럽트 신호가 정상이 아니다.
2. 인터럽트 신호가 엄청 자주 발생함.
-> 가끔 인터럽트 신호가 엄청 자주 발생해 시스템 오동작을 유발하기도 함. 이를 IRQ Storm 이라함.
So, 바로 인터럽트 파형 측정 하는 게 좋음.

먼저 파형은 인터럽트를 인지하는 방식에 따라 크게 에지 트리거와 레벨 트리거로 분류됨.

- 전기신호가 다른 상태로 넘어가는 순간 인터럽트 식별
2. 레벨 트리거
- 인터럽트 라인을 하이로 유지할 때 인터럽트 식별

-> 이미 커널에서는 인터럽트 신호를 식별하는 방식에 따라 인터럽트를 설정하도록 플래그를 정의해둠.

인터럽트 후반부 처리

-> 인터럽트는 빨리 처리되어야 하는데 만약 처리할 일이 많다면?? 2단계로 나누어 전반부먼저 처리하고 이후 후반부를 처리함. 네트워크 패킷 통신이나 멀티미디어를 처리하는 드라이버에서 인터럽트 후반부 기법 많이 적용됨. So, IRQ 스레드, Soft IRQ, 테스크릿과 같은 인터럽트 후반부 기법 잘 알아야함.

# 4. IRQ 스레드

1. IRQ 스레드
: Interrupt Request. 인터럽트 핸들러에서는 바로 처리하지 않아도 되는 일을 수행하는 프로세스.
: 확인 법
-> ps -ely : irq/86-mmc : irq/인터럽트 번호-인터럽트 이름
: irq 스레드는 라즈비안에서는 1개뿐. 다른 리눅스레서는 여러개인데 이게 많다고 좋은 게 아니라 어떻게 설계했느냐가 더 중요.

1.1 how 생성
: 부팅 과정에서 한 번 생성.
: request_threaded_irq-> setup_irq_thread -> kthread_create

1.2 request_threaded_irq
: 1단계 : 인자로 들어온 값을 인터럽트 디스크립터로 할당함
2단계 : irq_handler_t 의 thread_fn인자에 irq스레드 처리 함수 주소를 지정하면 irq스레드가 생성됨.

* irq 스레드 처리함수 : 인터럽트 별로 지정한 irq 스레드 별로 후반부를 처리하는 함수
* irq 스레드 핸들러 함수 : irq_thread 함수를 뜻하며 인터럽트 별로 지정된 irq 스레드 처리함수를 호출하는 역할.
