# 1. Task_struct
: 프로세스를 생성(fork)하든 쓰레드를 생성(clone)하든 task structure 가 하나 생성됨. 프로세스가 생성된 것은 결국 스레드도 하나 생긴다는 의미.

리눅스는 1대1 모델 기반으로, 사용자 모드에서 생성된 쓰레드를 커널도 암

리눅스에서는 프로세스든 쓰레드든 커널내부에서는 task라는 객체로 관리함.
Task가 관리하는 자원을 어떻게 공유하고 접근 제어하느냐에 따라 프로세스로 해석되기도 쓰레드로 해석되기도 함.
Fork, clone 모두 커널내부에서 마지막으로 호출되는 함수는 do_fork()이다. 이때, 이 함수의 인자로 보모 task와 얼마나 공유할지를 정해줌으로써 fork와 clone함수 둘 다를 지원할 수 있는 것이다.
( fork는 부모 테스크와 공유 적음, clone은 많음)

Do fork가 하는 일은 우선, 태어난 시간, 부모님이름, 소지품 등 자세한 정보를 저장하는 이름표를 만드는데 이것이 바로 task struct이다.

Posix표준에 의해 한 프로세스 내의 쓰레드는 동일한 pid를 공유해야한다. 따라서 리눅스는 이를 위해tgid 개념을 도입함.

1. Tgid의 정의

Thread Group ID는 프로세스의 리더 스레드(메인 쓰레드)의 **PID(Process ID)**와 동일합니다.
같은 프로세스에서 생성된 모든 스레드는 동일한 Tgid를 공유합니다.
즉, Tgid는 스레드가 속한 프로세스 그룹을 나타내는 ID입니다.

Clone함수의 인자로 clone child cleared와 clone child settid 를 설정하면 자원 공유가 되지 않는 프로세스를 생성하고 인자로 clone thread 를 설정 시 자원공유가 되는 쓰레드를 생성함.

실제 task struct 내의 tgid 값을 출력하는 함수는 getpid() 이고, pid값을 출력하는 함수는 gettid이다.

Asmlinkage 지시어로 보통 시스템콜 함수가 구현되어있는 데 이는 어셈블리로 작성된 파일에서 호출될 수 있도록 하였다.

테스크에는 훨씬 많은 정보가 사실 필요함. 우선순위, cpu사용량 등 이러한 모둔 정보를 context라고함

테스크의 문맥(context)은 크게 3개이다

첫째 시스템문맥으로 테스크의 정보를 유지하기 위해 커널이 할당한 자료구조들이다. 대표적인 자료구조로는 task struct, fd, 파일 테이블, 세그먼트 테이블, 페이지 테이블 존재.

둘째 메모리 문맥으로 텍스트, 데이터, 스택, 힙, 스왑공간이다

셋째 하드웨어 문맥으로 문맥교환할 때 테스크의 현재 실행 위치에 대한 정보를 유지하며 쓰레드 구조 또는 하드웨어 레지스터문맥으로 불린다.¹

Task struct 속 각 변수명
- task identification : 테스크 인식용 변수 pid, tid등
- state : 테스크의 생성에서 소멸까지 많은 상태를 거치며 이를 관리하는 변수.
- task relationship: 테스크는 생성되면서 가족관계를 갖음.
- scheduling information: 스케줄링과 관련된 변수
- signal information: 테스크에게 비동기적린 사건의 발생을 알리는 매커니즘
- memory information: 테스크는 자신의 명령어와 데이터를 텍스트, 데이터, 스택 ,힙에 저장하는데 이를 관리하는 변수
- file information: Task_struct 에서 files_struct 구조체 형태인 files라는 이름의 변수로 접근 가능.
- thread structure : 쓰레드 구조는 문맥 교환을 수행할 때 테스크가 현재 어디까지 실행되었는지 기억공간
- time information: 테스크의 시작시간 및 사용한 cpu 시간
- format :
- resource limits : 테스크가 사용할 수 있는 자원의 한계

테스크 상태 전이
1. 준비상태
2. 실행상태
-> 실행상태(running)에서 할 일을 다하고 exit 호출 시 exit zombie ( cpu에게 자원 반납)상태로 전이.
-> 실행상태에서 실제 수행하돈 테스크가 자신에게 주어진 cpu 시간을 모두 사용하면 준비(ready)상태로 전이.
-> 실행상테에서 테스크가 특정한 사건을 기다려야 할 필요가 있으면 대기 상태(task interrupitible)로 전이. Ex) lock. 이때 자신이 기다리는 동안 기다리는 사건 외에 일체 방해받아선 안되는 매우 중요한 경우에는 task uninterruptible, 그렇지 않으면 task interrupitlble).
스케쥴러는 대기 중이던 테스크를 실행상태로 만들고 그러다 테스크가 기다리던 event가 발생하면 준비상태로 전이하여 다른 태스크와 경쟁하게 됨.

실행상태는 다시 사용자 수준 실행 상태와 커널 수준 실행상태로 구분됨. 커널 수준 실행이 더 강력하다. 사용자 수준 실행 상태가 커널 수준상태로 전이되는 방법은 2가지 존재.
첫째, 시스템 호출의 사용.
둘째, 인터럽트의 발생.

사용자 수준 수행에서 스택관리는 0~4GB 까지의 주소 공간 중 3GB 아래 부분에 존재하는 스택 사용.
커널 수준 상태는 리눅스 그 자체로, 이는 스택을 필요로 함. 결론적으로 리눅스는 테스크가 생성되면 task struct 과 8kb 의 스택을 할당함. 이때 이 스택을 thread union이라 불림.
만약 테스크가 시스템 호출로 커널 수준상태로 진입한 뒤, 할일을 끝내면 다시 사용자 수준상태로 복귀하여 수행하던 곳에서부터 다시 작업을 해야함으로 이때 작업 위치 저장공간을 pt regs 에 저장한다.
Ex) open() 호출 시, 이 시스템 호출의 번호는 unistd.h 라는 헤더파일에 정해져있고 인텔 cpu 시 eax레지스터에 저장됨. Open이라는 시스템콜의 핸들러인 sys open 은 세가지 인수를 받는데 각각 ebx, ecx, edx 에 저장. 결국 시스템 호출 도중 이 레지스터둘의 내용이 바뀌게 되며 따라서 원래의 레지스터 내용을 pt regs 구조체에 담아 8kb 스택내에 저장함.

# 2. 스케줄링

- 스케줄러는 수행가능한 상태의 테스크를 큐 자료구조로 연결함 . 이를 런큐 라고 불림. 테스크가 생성되면 init task를 헤드로 이중 연결리스트에 삽입됨. 이를 task list 라고하겠다. Wake up new task 는 새로 생성되어 task running 상태가 된 테스크가 런 큐로 삽입되는 과정을 보여주며, wake up 함수는 이벤트를 대기하다 깨어나서 런 큐로 삽입되는 과정을 보여줌.
다중 cpu 환경에서 런큐가 여러 개라면 task list 에서 런큐로 삽입될 때 새로생성되는 테스크는 부모 테스크가 존재하던 런큐로 갑입된다ㅡ 이는 자식과 부모가 같은 cpu에서 실행될 때 더 높은 캐시 친화력( cache affinitiy)를 얻을 수 있기 때문. 대기 상태에서 깨어난 테스크는 대기 전에 수행되던 cpu의 런큐로 삽입됨 이도 같은 이유. 구체적으론 task struct 의 cpus allowed 필드에 저장된 cpu 번호로 결정함.

스케줄러는 cpu 의 런큐에서 다음번 수행시킬 테스크를 고름. 이때 2가지 고려사항.
첫째, 가장 높은 우선순위의 테스크를 선택하기 의해 필요한 부하를 어떻게 줄일 수 있나?
이를 위해 뒤에서 O(1) 스케줄러 제공함
둘 째, 런 큐가 비어있는 동안 어떻게 할 것 인가?
이를 위해 리눅스는 부하 균등 (load balancing) 기법을 제공함. 이 load balancing 함수에서는 특정 cpu가 한가하면 다른 바쁜 곳으로 이주시켜서 성능향상시킴.

사실 위 설명은 SMP 구조의 시스템일때 해당됨.
Symmetric Multi Processor로써 각 cpu가 동등하기 때문에 테스크가 어느 곳에서든 수행될 수 있음. 한편 NUMA(non uniform memory access)구조 에선 load balance 함수에서 cpu 부하뿐만 아니라 메모리 접근 시간의 차이 도 고려하여 부하 균등을 시도함.

스케줄러는 언제 어떻게 호출되나?
어떻게에 대한 답은 2가지
첫째, 직접적으로 schedule()함수 호출
둘 째, 현재 수행되는 테스크의 thread info 구조체내부에 존재하는 flags 필드 중 need resched 필드를 설정하는 방법.

언제에 대한 방법은 여러가지.
1. 주기적으로 클럭 인터럽트가 발생하는데 이게 종료되는 시점에 현재 수행중인 테스크의 need resched 필드를 보고 스케줄링할 필요가 있다면
2. 새로 생성된 테스크가 현재 수행되는 테스크를 선점해야한다면.
3. 현재 수행중인 테스크가 자신의 타임슬라이스를 모두 사용한 때.
4. 현재 테스크가 schedule setscheduler 같은 시스템 콜 호출 시.

커널 스케줄러의 동작원리?
- task struct 는 policy, priority, rt pri6, time slice 등 필드 존재. Policy 는 이 테스크가 어떤 스케쥴링 정책을 사용하는지 나타내고 fifo, rr , 등 사용가능. Fifo, rr 는 실시간 스케쥴링 정책이고 ,others 는 비실시간 정책임.
실시간 종책은 rt priority 는 0~99까지의 우선순위를 가질 수 있고 테스크가 수행을 종료하거나 중지하거나 cpu시간을 다 쓸 때 까지 cpu사용함. 또한 이들은 고정 우선순위가 지급되며 항상 높은게 낮은것보다 먼저 수행됨.

비실시간 정책은 -20~19사이의 우선순위를 갖게 되며 동적으로 변함. 이값은 priority 필드에 저장. 한편 커널에서 우선순위 계산시 priority + 120을 하여 사용.

시그널은 테스크에게 비동기적인 사건의 발생일 알리는 역할 . 시그널을 원활히 사용하려면 우선 다른 테스크에게 시그널을 보낼 수 있어야하고 이를 위해 리눅스는 sys kill 이라는 시스템 호출을 제공함. 둘째, 자신에게 시그널이 오면 그 시그널을 수신할 수 있어야하며 이를 위해 task struct 에선 signal, pending 이라는 변수가 존재함. 셋째 자신에게 시그널이 오면 그 시그널을 처리할 수 있는 함수를 지정해야한다. 이를 위해 sys signal 시스템콜 존재하며 task struct 내에 sighand 변수 존재

테스크들끼리 통신하는 기법이 IPC이다. 알려진 통신기법으로는 시그널, FIFO, 파이프, 메세지큐

리눅스는 시그널을 두가지로 구분하는데 일반 시그널 32개와 posix 표준을 위해 도입한 실시간 시그널 32개이다. 리눅스가 지원하는 시그널 목록은 kernel/include/asm arch/singal.h 에서 확인 가능.
Kill pid 시 pid는 실제로 tgid를 의미하고 같은 쓰레드들이 종료되려면 이들 간에 시그널을 공유하는 메커니즘이 필요함. 이러한 시그널을 보내는 작업은 sys kill 의 시스템호출에서 이뤄지고 시그널은 task struct 의 signal 필드에 저장함.
But 특정 테스크에만 보내야할 땐 공유하지 않는 시그널은 pending필드에 저장함.

각 테스크는 특정 시그널이 수행될 때 발생할 함수 즉 시그널 핸들러를 지정가능. 사용자 지정 시그널 핸들러를 설정하는 함수가 sys signal 이다.이는 task struct 에 sighand 필드에 저장됨.

또한 테스크는 특정 시그널을 받지 않도록 설정하려면 blocked 필드를 통해 이뤄짐. But, sigkill, sigstop은 무시못함 .

결론 : 우선 해당 테스크의 task struct 구조체를 찾고 보내려는 시그널 번호로 siginfo 자료구조를 초기화하고 시그널의 성격에 따라 signal 이나 pending 필드에 매달아준다. 이때 block 필드를 검사하여 확인한다. 인터럽트와 트랩은 커널에게 사건의 발생을 알리고 시그널은 task 에게 알림.

위 O(1)의 문제점(시간할당시문제) 를 해결하고자 cfs 등장. Cfs 는 other을 제외하고 sched normal, batch, idle 정책과 실시간 정책인 rr fifo 제공.

Cfs는 그룹 스케줄링을 지원하는데 사용자 id기반 그룹 스케줄링과 group 가상 파일시스템 기반 스케줄링으로 나뉨.
Cfs의 런큐는 cfs rq , rt rq 라는 자료구조를 갖고 rt rq 는 실시간 스케쥴링, cfs rq는 비실시간 자료구조임.

# 6. New 시스템 호출 구현방법
- sys newsyscall 이라는 시스템 호출 구현
1. 리눅스 커널에서 각 시스템 호출마다 유일하다고 했던 시스템 호출 번호를 하나 새로 할당함. 위 unistd_32.h 파일에 325개의 호출 번호가 할당되어있는데, newsyscall를 325번에 할당한다. #define___NR_newsyscall 325.

3. 시스템 호출 처리 함수 구현. 시스템 호출 처리 함수들 중에 테스크 관리자와 관련된 함수들은 리눅스 커널 소스 트리 중 kernel/ 디렉터리에, 파일시스템과 관련된 함수들은 fs/ 디렉터리에 구현하는 것이 일반적. 이 예에서는 kernel/ 디렉터리에 newfile.c 라는 이름의 파일로 구현해봄. Vi 편집기로 아래와같이 작성.
#include linux unistd.h, errno, kernel, sched.h

{
printk("hello linux")

EXPORT_STMBOL_GPL(sys_newsyscall)

이 모듈은 hooking module init에서 시ㅓㅡ스템호출 테이블에 이미 등록된 sys open 시스템 호출을 우리가 구현한 sys our open으로 대치함. 이를 위해선 메모리상네 존재하는 sys call table 의 주소를 알아야하는데 예전에는 extern을 사용하여 커널에서 그 주소를 가져다 사용했었다.
구체적으론 Extern void *sys call table[] 과 같이 작성하여 커널이 가지고 있는 젼수 중 sys call table 의 주소를 현재 커널에 삽입되는 모듈에서 참조하여 사용했었다. 이때 리눅스 커널은 외부에서 sys call table이라는 심볼을 참조할 수 있도록 허락해 둔 상태여야 함. But 리눅스개발바들은 외부에서 보안상의 이유로 sys call table을 참조하지 못하도록 해놓았다. 따라서 EXPORT SYMBOL 매크로로 sys call table의 외부에서 참조될 수 있도록 만들어 주어야한다. But 이 방식은 상당히 귀찮음. 따라서 여기선 locate sys call table 함수를 작성하여 커널이 존재하고 있는 가상 주소 공간을 뒤져서 sys call table이 가지고 있을 것으로 예상할 수 있는 값이 프로그램에서는 커널의 시적 위치(0xC000 0000) 부터 4BYTE씩 증가 시키면서(for문) 배열의 _NR_close 위치에 sys close 함수의 주소가 존재하는 것을 찾으며, 결국 이 배열의 위치가 sys call table의 위치가 된다. ( 일부 커널에선 이 프로그램이 기존 커널소스와 충돌을 일으키는 데 이땐 sys call table변수이름과 locate sys call table 이름을 다른 것으로 변경시켜주어야 함)

또한 리눅스가 sys call table의 page table을 write가 안되도록 설정해 놓았기 때문에 직접 페이지 테이블 엔트리을 찾아내어 write가 가능하도록 _PAGE KERNEL 속성을 지정해주었다. 다음으로 페이지 테이블 엔트리를 수정한 뒤 곧바로 접근하게 되면 에러가 발생하므로 커널 내에 구현된 __flush tlb single 매크로를 사용하여 tlb내용을 flush 해주었다.

이렇게 확인한 sys call table에서 sys open 에 대한 함수 포인터가 존재하는 위치를 알기 위해 __NR_open 이라는 상수를 사용했다. 그런 뒤 sys open 을 가리키고 있던 포인터를 우리가 모듈 내에서 구현한 sys our table 을 가리키도록 수정함. 그러면 사용자가 open함수를 요청할 때 리눅스 커널에서 sys open이 아닌 sys our open이 호출됨. Sys our open은 sys open 을 호출함. 따라서 open을 호출한 프로그래머입장에선 커널이 이전처럼 자신의 요청을 잘 서비스 해준 것처럼 느낌. 단지 시스템 콜의 실제 핸들러로 진입하기 전에 이를 가로채는 것으로써 우리는 원하는 정보를 얻을 수 있게 된다.

결국 이 모듈로 어떤 사용자가 어떤 파일을 오픈했는지 추적가능함. 직원들 근무 잘하는 지 확인 시 유용.

# 7. 디바이스 드라이버

- 유닉스 계열에서 모둔 것을 파일로 취급함. 지금까진 한 것은 정규파일, 장치들에 대한 파일은 장치파일이라고 부른다. 사용자 테스크가 file operation구조체에 정의 되어 있는 함수를 통해 장치파일에 접근할 때 호출할 함수를 정의하고 구현해주는 것이 디바이스 드라이버이다. 디바이스 드라이버를 구분하기 위해 주 번호 0부터 255까지 지원함. 이는 I node 객체에 주번호가 기록되어있다. 이때 부번호는 20bit를 차지한다. 부번호는 같은 디바이스 드라이버를 사용하는 장치가 복수 개 있을 때 이를 구분하기 위해 사용된다.
Ex) 모니터는 /dev/ tty 란 이름으로 /dev/tty0 4,0이 하나의 모니터이다. 이때 tty0,1,2,3은 서로 다른 모니터이고 4는 모니터를 지칭하는 주번호이고 0,1,2,3 은 부번호이다.

또한 한개의 물리 디스크에 지원하는 최대 파티션 개수는 64개이다.리눅스는 디바이스 드라이버를 크게 문자 디바이스 드라이버, 블록 디바이스 드라이버, 네트워크 디바이스 드라이버로 세 개로 구분함.
1. 문자 디바이스 드라이버
- 순차 접근이 가능하고 임의의 크기로 데이터 전송이 가능한 드라이버로 터미널 드라이버가 있다.
2. 블록 디바이스 드라이버
- 임의 접근이 가능하고 고정된 크기의 블록 단위로 데이터를 전송하는 드라이버로 디스크 드라이버가 있다. 하지만 최근에는 나누지 않고 또 수십가지 종류가 있다. 현재는 단지 커널 내부의 버퍼캐시와 큐를 통해 데이터를 주고받는가의 여부에 따라 문자와 블록 디바이스 드라이버를 구분함.
사용자의 read, write함수와 1:1로 매칭되어 디바이스 다라이버의 함수가 호출된다면 이를 문자 디바이스 드라이버라고 함.
블록 디바이스 드라이버는 사용자의 read, write와 같은 함수에 대응되는 함수가 존재하지 않으며, 큐를 통해 버퍼캐시와 통신함.

디바이스는 어떤 os환경이든 아니든 동작하기 때문에 디바이스 드라이버도 어느 os든 동작하도록 만들어야하는데 이는 쉽지 않다. 따라서 디바이스 드라이버를 작성할 때 디바이스를 구동시키기위해 필요한 하드웨어와 밀접한 코드와 운영체제와 관련된 코드를 분리하는 것이 바람직하다. 따라서 리눅스의 디바이스 드라이버는 특정 하드웨어를 위한 디바이스 드라이버 코어와 코어를 리눅스에서 사용가능한 형태로 만들어 주기 위한 일종의 래퍼로 구성됨.

코어는 하드웨어 메뉴얼에 따라 작성하고 이를 리눅스 커널에 등록하고 사용자 테스크가 장치 파일을 통해 접근할 수 있게 해주려면 사용자 테스크가 호출할 함수들과 코어의 함수를 연결시켜줘야 한다. 이것이 바로 래퍼의 역할이다. 그런데 어떤 함수를 사용자 테스크에게 제공해 줄 것인지 고민할 필요가 있을까? 그렇지않다. 왜냐면 리눅스 커널은 드라이버의 래퍼가 제공해야할 함수들을 미리 정의해 놓았다. 즉 디바이스 드라이버 개벌자는 어떤 함수를 서용자 테스크에게 제공해야 할지 고민없이 파일 오퍼레이션 자료구조에 이미 정의된 함수들에 대해서만 제공하면됨 . 결국 디바이스 개발이 용이해진다.

지금까지 사용자 테스크와 디바이스 드라이버 개발 입장에서 살펴보았다. 사용자 테스크는 특정 디바이스 드라이버가 어떤 함수를 제공하는지 일일이 알 피료없이 파일 오퍼레이션 자료 구조에 정의된 함수를 통해 일관되게 장치에 접근할 수 있다. 또한 디바이스 드라이버 개발자는 어떤 함수를 사용자 테스크에게 제공해야 할지 고민할 필요없이 파일 오퍼레이션 구조체만을 고려하면 됨. 이것이 리눅스가 채택하는 디바이스 그라이버 관리 구조임.

사용자 테스크는 시스템 호출로 장치파일에 접근한다. 이때 호출 가능한 함수는 파일 오퍼레이샨 두조체에 정의 되어있다. 반대로 디바이스 드라이버 개발자는 ddi(device driver interface), 즉 파일 오퍼레이선 구조체에 정의되어 있는 함수를 디바이스 드라이버 내에 구현해줌으로써 인터페이스 제공함.

# 8. 네트워킹

대표적인 소켓 인터페이스는 socket, bind, connect, accept, listen, send, recv 존재

Socket : 프로토콜을 선택하는 인터페이스
Bind : 자신의 호스트주소와 응용을 선택하는 인터페이스
Connect : 목적지 호스트 주소와 응용을 선택하는 인터페이스
Accept : 서버와 클라이언트 통신 요청을 받아들일 때 사용
Listen : 서버가 동시에 몇 개의 요청을 서비스 하겠다 결정시 사용
연결 설정 이후 send, recv 또는 read write로 전송 가능.

통싱 프로토콜에 사용하는 자료구조로는 bsd소켓 층에선 socket 자료구조, tcp ip 전 층에서 사용되는 sock 자료구조, 디바이스 층의 net device 자료구조 존재.

- 컴파일러, 어셈블러, 링커, sdk 등
- 소스코드 .c --컴파일러-->어셈블리어--어셈블러-->(기계어) 목적파일.o --링커-->실행파일.out .// 링커에는 sdk 즉 헤더파일, 라이브러리 등이 목적파일과 결합하는 과정임.

- gcc -c *.c : 소스코드를 목적파일.o로 변환
- ld *.o : 링킹 //그냥 하면 에러남-> 라이브러리 설정 및 c runtime 목적파일이 없어서
- C runtime : main 함수에 이르기까지 초기화 수행

컴파일은 상당히 오래 걸리는 작업이므로 컴파일과 링킹을 나누어서 원하는 파일만 컴파일하고 빌드하는 점진적 빌드를 수행하는 것이 효율적.(=> 수정된 파일만 컴파일 가능-> 전체 빌드 시간 단축) But 개발자가 모든 파일들의 상호관계를 알아야함 .So 빌드 시스템인 makefile, CMake, Ninja가 존재하여 의존성을 파악하지 않아도 점진적빌드가 가능하게 도움.

[구조]
타겟 : 의존성
명령어 ~~

타겟: 생성할 것
의존성 : 필요한 것
명령어 : 해야하는 것.
단, 한 줄이 하나의 프로세스
Ex)
main.out : main.o utils.o
gcc main.o utils.o -o main.out

==>main.out 을 만들기위해 main.o, utils.o 파일이 필요하다.

Kconfig 는 커널 모듈이나 기능을 수성하는 데 필요한 성정을 정의하는 파일로 , menu ~ endmenu로 끝남. 이는 make menuconfig에 의해 읽혀짐.

If) main2.c 라는 소스코드를 추가했다면 Makefile에 추가해줘야함. 그래야지만 make로 컴파일 할때 같이 포함해서 컴파일함. 안적으면 그냥 무시해버림.

gcc 컴파일 : 항상 전체 빌드. 파일 하나일때 사용
Make 컴파일 : Makefile 로 정의되며 점진적 빌드 가능. 대규모 파일할 때 효율적.
